## Design decision record (2026-07-11)

The original backlog specced privacy-by-default: `entries.visibility` with a four-level enum and RLS gating who can see whole entries. We challenged this and landed on a different model:

**Ratings are network signal; notes are circle candor.**

- **Ratings** (status + overall + factor ratings) are visible to all members. They feed the trust engine (v0.6+) and score traceability (Pandora-not-Spotify) — a score you can decompose requires contributions you can see. Strangers' ratings reach you through place cards and score breakdowns, **never through a browsable stranger feed** (Inner-Circle Integrity).
- **Notes and comments** are circle-scoped. Candor is the data moat — the 4.5-star McDonald's problem exists because Google's raters perform for a public audience. People write "overrated, service was rude" only when strangers, owners, and employers can't read it.
- **No per-entry private flag** in 0.4.0. One mental model. Adding privacy later breaks no promises; removing it later would.
- **Friendship's job is weighting + candor scope, not access control** over ratings.

Consequences for the backlog: items …658 (visibility RLS), …659 (visibility selector), …660 (filter chips) need rewording — see "Backlog updates" at the bottom.

> **Concept: why the schema splits — RLS is *row*-level.**
> Postgres Row-Level Security decides whether you get a row, not which columns of it. Since ratings (member-visible) and notes (circle-only) now have different audiences but live on the same `entries` row, the private part must become its own row: a new `entry_notes` table carrying the free-text fields, with its own circle-scoped policy. A product boundary becoming a schema boundary — the same reason entitlement tiers in market-data products end up as separate tables, not columns.

## The big picture

1. A **friendships** table records the graph (Phase 1).
2. **UI** to find friends, send/accept requests, browse the circle (Phases 2–3).
3. **Notes split out** of `entries` into `entry_notes`; notes + comments get circle-scoped RLS (Phase 4).
4. **Feed chips** become relevance lenses — My Circle / Just Mine / Everyone (Phase 5).
5. **Place tags** — multi-author classification replacing binary `place_type` (Phase 6).

## Data model

### Friendships: two directional rows per friendship

```
Alice requests Bob:
  (user_id=alice, friend_id=bob,   status='pending')
  (user_id=bob,   friend_id=alice, status='pending')
Bob accepts → both rows flip to 'accepted'.
```

Two rows so every RLS check and UI query is "rows where `user_id = me`" — no OR-gymnastics — and block/remove can be asymmetric later. The pair is kept consistent by doing **all writes through RPCs**, never raw inserts. A `requested_by uuid` column distinguishes incoming from outgoing pending requests.

**Statuses:** `pending`, `accepted`, `blocked`, `removed`.

### All friendship writes go through RPCs

Alice sending Bob a request must create a row with `user_id = bob`, which RLS correctly forbids her from inserting directly. Supabase's answer: `SECURITY DEFINER` Postgres functions called via `supabase.rpc(...)`:

- `send_friend_request(target uuid)`
- `respond_friend_request(other uuid, accept boolean)`
- `cancel_friend_request(target uuid)`
- `remove_friend(target uuid)` / `block_user(target uuid)` / `unblock_user(target uuid)`

> **Concept: SECURITY DEFINER** — runs with the *creator's* privileges, not the caller's: a bank teller who can touch the vault, but only to execute your one validated transaction. All state-machine rules (no self-friending, no double-accept, blocked pairs can't re-request) live inside these functions.

### entry_notes: the circle-scoped half of an entry

```sql
CREATE TABLE public.entry_notes (
  entry_id   uuid PRIMARY KEY REFERENCES public.entries (id) ON DELETE CASCADE,
  user_id    uuid NOT NULL REFERENCES public.users (id) ON DELETE CASCADE,  -- denormalized author for cheap RLS
  notes      text,
  try_note   text,
  updated_at timestamptz NOT NULL DEFAULT now()
);
```

`entries` keeps: status, all five ratings, url, timestamps — member-visible, RLS unchanged. `notes` and `try_note` migrate to `entry_notes`, then drop from `entries`. `user_id` is denormalized onto the notes row so the RLS policy doesn't need a join.

### Helper to avoid RLS recursion

```sql
CREATE FUNCTION public.is_accepted_friend(a uuid, b uuid)
RETURNS boolean LANGUAGE sql SECURITY DEFINER STABLE ...
```

Used by `entry_notes` and `comments` policies — a `SECURITY DEFINER` lookup avoids recursively evaluating `friendships`' own RLS inside another table's policy.

### Email search without exposing emails

`public.users` has no email column (email lives in `auth.users` — good). So:

- `users.allow_email_lookup boolean NOT NULL DEFAULT false` — opt-in.
- RPC `find_user_by_email(email text)` — exact match only, only if the target opted in, never returns the email itself. No partial match (no scraping).

Handle search is a plain `ilike` on `users.handle` (already unique, already readable).

### Small additions

- `users.bio text` — friend profile page needs it; column doesn't exist yet.

## Phase 1 — Migration: friendships table + RPCs

**New file:** `supabase/migrations/0017_friendships.sql`

1. ```sql
   CREATE TABLE public.friendships (
     user_id      uuid NOT NULL REFERENCES public.users (id) ON DELETE CASCADE,
     friend_id    uuid NOT NULL REFERENCES public.users (id) ON DELETE CASCADE,
     status       text NOT NULL CHECK (status IN ('pending','accepted','blocked','removed')),
     requested_by uuid NOT NULL REFERENCES public.users (id),
     created_at   timestamptz NOT NULL DEFAULT now(),
     accepted_at  timestamptz,
     PRIMARY KEY (user_id, friend_id),
     CHECK (user_id <> friend_id)
   );
   ```
2. `is_accepted_friend(a, b)` helper.
3. The six RPCs, each enforcing the state machine (`pending → accepted | removed`; `blocked` exits only via `unblock`; blocked pairs can't re-request).
4. RLS: SELECT own edges only (`user_id = auth.uid()`). **No INSERT/UPDATE/DELETE policies** — writes go through RPCs exclusively.
5. Add table to the Realtime publication (Phase 3 badge needs it).
6. `users` changes: `bio`, `allow_email_lookup`, `find_user_by_email` RPC.

**Acceptance check:** run migration locally; as two test users exercise the full state machine in the SQL editor — request, accept, remove, re-request, block, verify a blocked user's `send_friend_request` raises, verify direct `INSERT INTO friendships` is rejected.

## Phase 4 — Migration: entry_notes split + circle-scoped RLS

**New file:** `supabase/migrations/0018_entry_notes_and_comment_privacy.sql`

1. Create `entry_notes` (schema above); backfill from `entries.notes` / `entries.try_note` (rows with at least one non-null field); drop those columns from `entries`.
2. RLS on `entry_notes` — the heart of the release:
   ```sql
   CREATE POLICY "notes visible to author and circle" ON public.entry_notes
     FOR SELECT TO authenticated USING (
       user_id = auth.uid()
       OR public.is_accepted_friend(auth.uid(), user_id)
     );
   ```
   INSERT/UPDATE/DELETE: author only (`user_id = auth.uid()`).
3. `comments`: add `quoted_user_id uuid REFERENCES public.users (id)` and backfill it from `quoted_comment_id` (the existing `quoted_author` is display-name text and the FK nulls on delete — neither reliably identifies the quoted *user* for a policy check). Then replace the read-all SELECT policy with the **fail-closed both-circles predicate** (decision 2026-07-11):
   ```sql
   CREATE POLICY "comments visible within both circles" ON public.comments
     FOR SELECT TO authenticated USING (
       (user_id = auth.uid() OR public.is_accepted_friend(auth.uid(), user_id))
       AND (quoted_user_id IS NULL
            OR quoted_user_id = auth.uid()
            OR public.is_accepted_friend(auth.uid(), quoted_user_id))
     );
   ```
   A quoting comment is visible only if the viewer is in the quoter's circle AND the quoted author's circle — the quote snapshot can never leak a stranger's words. Fails closed: a policy mistake hides too much rather than leaking. (Upgrade path if hidden replies feel too aggressive in practice: a `comments_visible` masking view that redacts `quoted_text` instead of hiding the row.) Note we can't derive the quoted author by subquerying `comments` inside its own policy — that's recursive policy evaluation, which Postgres rejects; hence the denormalized `quoted_user_id`. Same author-or-circle predicate for `comment_reactions` (via the reaction's author).
4. **Backfill mutual friendships among existing alpha users** (per cutover decision).
5. `entries` RLS: **unchanged** — ratings stay member-visible.

**Acceptance check:** three users — A/B friends, C stranger. A writes an entry with a note and a comment. B sees rating + note + comment. C sees the rating on the place card but no note and no comment — verify with raw Supabase queries, not just the UI. Verify C cannot fetch `entry_notes` rows directly via the API.

> Add to the acceptance matrix: A quotes B's comment (A/B friends); viewer C is friends with A but not B → C must not see A's quoting comment at all (fail-closed). D, friends with both, sees it fully.

## Phase 5 — UI: write path split + feed lenses

**Touches:** `index.html`, `src/js/app.js`, `src/js/ui-render.js`, `src/js/places-service.js`

1. **Write path**: the entry upsert in `places-service.js` splits — ratings/status upsert to `entries`, note text upserts to `entry_notes` (delete the row when the note is emptied). Load path: fetch both tables; RLS silently trims `entry_notes` to what you may see; merge on `entry_id` in JS.
2. **Filter chips** (`index.html` ~lines 82–86): `Everyone / Just mine` becomes **My Circle / Just Mine / Everyone**, default **My Circle**. These are *lenses, not walls* — client-side classification over already-fetched data (mirror the status-filter pattern at `ui-render.js:19`):
   - *My Circle*: takes by me or accepted friends
   - *Just Mine*: my takes
   - *Everyone*: all members' takes (ratings; notes render only where RLS returned them)
   Friends list for classification comes from `friends-service.js`, cached in app state at load.
3. **No visibility selector** in the add/edit modal. Instead, a small static hint next to the notes field: "Notes are visible to your friends only." Zero-decision UI.
4. Place cards render a take's note only when present in the merged data — absence is indistinguishable from "no note written," which is the correct privacy behavior.

**Acceptance check:** with A/B/C from Phase 4 — each chip shows exactly the right takes per user; default is My Circle; C's card for A's place shows A's rating but no note; a new entry's note round-trips through the split write path.

## Risks

- **RLS mistakes fail silently** — a wrong policy leaks notes or blanks them with no error. Mitigation: Phase 4's three-user matrix, tested via raw API queries.
- **Split write path** — entry saves now touch two tables; a failed second write leaves a rating without its note. Acceptable at alpha (retry on next edit); a Postgres function wrapping both writes is the clean fix if it bites.
- **Realtime publication** — forgetting to add `friendships` means the badge silently never updates.
- **RPC state-machine edge cases** — double-accept, request-after-remove, request-to-blocker; enumerate in Phase 1 SQL tests.
- **Quote-reply leakage** — mitigated fail-closed via the both-circles predicate on `quoted_user_id` (Phase 4.3). Residual risk is over-hiding, not leaking; revisit with the masking-view upgrade if threads feel gappy.
